## 1. Обзор проекта

### 1.1. Назначение системы

Hockey Analytics Platform — это комплексная система для автоматизированного анализа хоккейных игроков на основе открытых данных NHL API с применением искусственного интеллекта для оценки потенциала, сильных и слабых сторон игроков.

### 1.2. Цели и задачи

**Основные цели:**
- Автоматический сбор и агрегация статистики игроков из NHL API
- Расчет продвинутых метрик (xG, Corsi, Fenwick, zone entries)
- AI-анализ игроков с оценкой потенциала развития
- Предоставление данных через веб-дашборд и чат-боты
- Интеграция с внешними системами через стандартизированные API

**Ключевые функции:**
1. Поиск игроков по имени с автодополнением
2. Просмотр детальной статистики за сезон/карьеру
3. AI-генерация аналитических отчетов
4. Сравнение игроков между собой
5. Прогнозирование развития карьеры
6. Уведомления о значимых событиях (травмы, контракты)

### 1.3. Целевая аудитория

- Скауты хоккейных клубов
- Спортивные аналитики и журналисты
- Фанаты хоккея с углубленным интересом к статистике
- Агенты игроков
- Тренерский штаб

## 2. Архитектурные принципы

### 2.1. Основные принципы проектирования

**Микросервисная архитектура:**
- Каждый компонент независим и масштабируется отдельно
- Четкое разделение ответственности между сервисами
- Слабая связанность через стандартизированные интерфейсы

**Event-Driven подход:**
- Асинхронная обработка данных
- Событийная модель для триггеров анализа
- Очереди сообщений для надежной доставки

**API-First:**
- Все функции доступны через API
- RESTful принципы для HTTP-эндпоинтов
- MCP (Model Context Protocol) для AI-агентов

**Data-Centric:**
- PostgreSQL как единый источник правды
- JSONB для гибких схем данных
- Кэширование на нескольких уровнях

### 2.2. Технологический стек

| Компонент | Технология | Обоснование |
|-----------|------------|-------------|
| **Backend** | Rust (Tokio, Axum) | Производительность, безопасность памяти, async/await |
| **База данных** | PostgreSQL 15+ | JSONB, надежность, ACID |
| **ORM** | sqlx | Compile-time проверка SQL, async |
| **AI Оркестрация** | n8n | Визуальный workflow, вебхуки |
| **AI Модель** | OpenClaw | Локальный LLM, контроль данных |
| **MCP Сервер** | Rust + MCP SDK | Стандарт для AI-агентов |
| **Чат-бот** | Telegram Bot API | Массовость, простота интеграции |
| **Дашборд** | React/Vue + Rust API | Интерактивность, производительность |
| **Контейнеризация** | Docker + Docker Compose | Переносимость, простота развертывания |

**Level 1: Client Layer (Клиентский уровень)**
- Веб-интерфейс для аналитиков
- Чат-боты для мобильных пользователей
- MCP-клиенты для AI-агентов
- Внешние системы через вебхуки

**Level 2: Application Layer (Прикладной уровень)**
- API Gateway на Rust/Axum
- Бизнес-логика в виде сервисов
- MCP Server для AI-интеграций
- Middleware для безопасности и валидации

**Level 3: Data Layer (Уровень данных)**
- PostgreSQL как основное хранилище
- Кэширование в памяти (Redis опционально)
- Full-text search через PostgreSQL

**Level 4: Integration Layer (Интеграционный уровень)**
- NHL API для получения статистики
- n8n для оркестрации workflows
- OpenClaw для AI-анализа
- Внешние источники новостей

## 4. Компоненты системы

### 4.1. Rust Backend (API Gateway)

**Назначение:** Центральный компонент, обрабатывающий все входящие HTTP-запросы, координирующий работу сервисов и обеспечивающий интеграцию с базой данных.

**Технологии:**
- Rust (версия 1.75+)
- Фреймворк: Axum
- Runtime: Tokio (async)
- HTTP клиент: reqwest
- Сериализация: serde + serde_json
- Валидация: validator

**Основные модули:**

1. **HTTP Server (Axum)**
   - REST API эндпоинты
   - WebSocket/SSE для real-time обновлений
   - CORS настройки
   - Compression (gzip/br)

2. **Middleware Stack**
   - Authentication (JWT/OAuth2)
   - Rate Limiting (token bucket)
   - Request/Response logging
   - Error handling & recovery
   - Request ID tracking

3. **Player Service**
   - Поиск игроков (fuzzy search)
   - CRUD операции
   - Валидация данных
   - Кэширование частых запросов

4. **Statistics Service**
   - Сбор статистики из NHL API
   - Нормализация данных
   - Расчет производных метрик
   - Агрегация по периодам

5. **Analytics Engine**
   - Расчет продвинутых метрик:
     - Expected Goals (xG)
     - Corsi/Fenwick
     - Zone Entry/Exit
     - Shot Quality
     - Player Impact Estimate
   - Сравнительный анализ
   - Выявление паттернов

6. **NHL API Client**
   - Интеграция с api-web.nhle.com
   - Retry логика с exponential backoff
   - Rate limiting соблюдение
   - Response кэширование
   - Error handling

7. **Cache Manager**
   - In-memory кэш (Tokio Mutex)
   - TTL управление
   - Cache invalidation стратегии
   - Статистика hit/miss

8. **Notification Service**
   - Email уведомления
   - Webhook dispatch
   - Queue management

**Конфигурация:**
```yaml
server:
  host: 0.0.0.0
  port: 8080
  workers: 4
  
database:
  url: postgresql://user:pass@localhost:5432/hockey_analytics
  max_connections: 20
  min_connections: 5
  
cache:
  ttl_seconds: 300
  max_size_mb: 512
  
nhl_api:
  base_url: https://api-web.nhle.com/v1
  rate_limit_per_minute: 60
  timeout_seconds: 30
  
jwt:
  secret: ${JWT_SECRET}
  expiration_hours: 24
```

### 4.2. PostgreSQL Database

**Назначение:** Централизованное хранилище всех данных системы с поддержкой транзакций, индексов и полнотекстового поиска.

**Версия:** PostgreSQL 15+

**Основные возможности:**
- ACID транзакции
- JSONB для гибких схем
- Full-text search (tsvector/tsquery)
- Materialized views для агрегаций
- Row-level security (опционально)
- Partitioning для больших таблиц

**Таблицы (подробно в разделе 7):**
- `players` — профили игроков
- `player_season_stats` — статистика по сезонам
- `ai_analysis` — AI отчеты
- `news_events` — новости и события
- `system_config` — конфигурация
- `audit_log` — журнал аудита

**Индексы:**
- B-tree для точных поисков
- GIN для JSONB полей
- GIN для full-text search
- Partial indexes для частых запросов

**Оптимизация:**
- Connection pooling (sqlx)
- Prepared statements
- Query plan анализ
- VACUUM автонастройка

### 4.3. MCP Server

**Назначение:** Предоставление стандартизированного интерфейса для AI-агентов (Claude, GPT, локальные LLM) через Model Context Protocol.

**Технологии:**
- Rust
- MCP SDK
- Transport: Server-Sent Events (SSE)
- JSON-RPC 2.0 для сообщений

**Инструменты (Tools):**

1. **get_player**
   ```
   Описание: Получить базовую информацию и статистику игрока
   Параметры:
     - name: string (required) — имя игрока
     - season: string (optional) — сезон в формате "20232024"
     - include_advanced: boolean (optional, default: false)
   Возврат: JSON с данными игрока
   ```

2. **analyze_player**
   ```
   Описание: Запустить полный AI анализ игрока
   Параметры:
     - name: string (required)
     - focus: enum ["potential", "strengths", "comparison", "all"]
     - depth: enum ["quick", "standard", "deep"]
   Возврат: Структурированный отчет с сильными/слабыми сторонами
   ```

3. **compare_players**
   ```
   Описание: Сравнить двух или более игроков
   Параметры:
     - players: array[string] (required, min: 2, max: 5)
     - metrics: array[string] (optional)
     - season: string (optional)
   Возврат: Сравнительная таблица метрик
   ```

4. **search_players**
   ```
   Описание: Поиск игроков по имени с автодополнением
   Параметры:
     - query: string (required, min: 2 chars)
     - limit: integer (optional, default: 10, max: 50)
     - position: string (optional)
   Возврат: Список найденных игроков
   ```

5. **get_team_roster**
   ```
   Описание: Получить состав команды
   Параметры:
     - team_abbreviation: string (required)
     - season: string (optional)
   Возврат: Список игроков команды
   ```

6. **get_player_news**
   ```
   Описание: Получить новости о игроке
   Параметры:
     - player_name: string (required)
     - days_back: integer (optional, default: 30)
   Возврат: Список новостей с источниками
   ```

**Транспорт (SSE):**
- Endpoint: `/mcp/sse`
- Content-Type: `text/event-stream`
- Keep-alive: 30 секунд
- Reconnect: автоматический

**Аутентификация:**
- API Key в заголовке `Authorization: Bearer <key>`
- Rate limiting: 100 запросов/минуту на ключ

### 4.4. n8n Workflow Automation

**Назначение:** Визуальная оркестрация сложных бизнес-процессов, интеграция AI (OpenClaw), автоматизация периодических задач.

**Технологии:**
- n8n (self-hosted)
- Webhooks для триггеров
- HTTP Request nodes для интеграций
- Code nodes для кастомной логики

**Основные workflows:**

**Workflow 1: Player Analysis Pipeline**
```
Trigger: Webhook (POST /webhook/analyze-player)
  ↓
[HTTP Request] → Rust Backend: GET /api/players/search?name={{ $json.name }}
  ↓
[Code Node] → Extract player_id from response
  ↓
[HTTP Request] → Rust Backend: GET /api/players/{{ player_id }}/stats
  ↓
[PostgreSQL] → Fetch historical data
  ↓
[Code Node] → Prepare context for AI
  ↓
[HTTP Request] → OpenClaw API: POST /analyze
  ↓
[Code Node] → Parse AI response, extract insights
  ↓
[PostgreSQL] → Save analysis to ai_analysis table
  ↓
[Webhook] → Notify MCP Server (cache invalidation)
  ↓
[Response] → Return analysis to client
```

**Workflow 2: Daily Stats Update**
```
Trigger: Schedule (every day at 02:00 UTC)
  ↓
[HTTP Request] → NHL API: GET /schedule/yesterday
  ↓
[Loop] → For each game
  ↓
  [HTTP Request] → NHL API: GET /game/{{ game_id }}/boxscore
  ↓
  [PostgreSQL] → Update player stats
  ↓
  [Code Node] → Recalculate advanced metrics
  ↓
[PostgreSQL] → Batch update
  ↓
[Webhook] → Trigger AI re-analysis for top performers
  ↓
[Email] → Send daily summary to subscribers
```

**Workflow 3: News Monitoring**
```
Trigger: Schedule (every hour)
  ↓
[HTTP Request] → News API: Search for hockey players
  ↓
[Code Node] → Filter relevant news, extract player names
  ↓
[PostgreSQL] → Save to news_events table
  ↓
[HTTP Request] → Rust Backend: POST /api/notify
  ↓
[Telegram Bot] → Send alert to subscribed users
```

**Workflow 4: Potential Prediction**
```
Trigger: Schedule (weekly, Sunday 00:00 UTC)
  ↓
[PostgreSQL] → Fetch players under 25 years old
  ↓
[Loop] → For each player
  ↓
  [Code Node] → Calculate age curves, comparables
  ↓
  [HTTP Request] → OpenClaw: Predict potential
  ↓
  [PostgreSQL] → Update potential_score
  ↓
[Code Node] → Identify breakout candidates
  ↓
[Webhook] → Notify scouts
```

**Конфигурация n8n:**
- Port: 5678
- Webhook URL: `https://your-domain/webhook/`
- Execution mode: regular
- Concurrency: 10
- Database: PostgreSQL (shared)

### 4.5. OpenClaw AI Engine

**Назначение:** Локальная AI модель для генерации аналитических отчетов, оценки потенциала и выявления паттернов в данных.

**Технологии:**
- OpenClaw (локальный LLM)
- API совместимость с OpenAI
- Модель: Mixtral 8x7B или аналогичная
- Context window: 32k tokens

**Endpoints:**

1. **POST /analyze**
   ```json
   Request:
   {
     "player_data": {
       "name": "Connor McDavid",
       "age": 27,
       "position": "C",
       "seasons": [...],
       "advanced_stats": {...}
     },
     "analysis_type": "full",
     "language": "en"
   }
   
   Response:
   {
     "strengths": ["...", "..."],
     "weaknesses": ["...", "..."],
     "potential": {
       "ceiling": "elite #1 center",
       "timeline": "peak now, 3-5 years",
       "confidence": 0.92
     },
     "comparable_players": ["...", "..."],
     "development_areas": ["...", "..."]
   }
   ```

2. **POST /predict**
   ```json
   Request:
   {
     "player_history": [...],
     "age": 22,
     "target_age": 27
   }
   
   Response:
   {
     "projected_stats": {...},
     "probability_breakout": 0.78,
     "risk_factors": ["..."]
   }
   ```

3. **POST /compare**
   ```json
   Request:
   {
     "players": [
       {"name": "...", "stats": {...}},
       {"name": "...", "stats": {...}}
     ]
   }
   
   Response:
   {
     "comparison_matrix": {...},
     "winner_by_category": {...},
     "overall_assessment": "..."
   }
   ```

**Промпты (системные инструкции):**

```
System Prompt for Player Analysis:
"You are an expert hockey analyst with 20+ years of experience 
scouting NHL players. Analyze the provided player statistics 
and provide detailed, objective assessment.

Guidelines:
1. Base all claims on actual data, cite specific numbers
2. Compare to league averages and positional peers
3. Consider age, experience, and team context
4. Identify both obvious and subtle patterns
5. Be honest about weaknesses, not just strengths
6. Use professional hockey terminology
7. Provide actionable development recommendations

Output format: JSON with strict schema validation."
```

**Кэширование:**
- TTL: 7 дней для стабильных игроков
- TTL: 24 часа для молодых игроков
- Invalidated при новых играх

### 4.6. ChatBot (Telegram)

**Назначение:** Удобный мобильный интерфейс для быстрого доступа к аналитике через мессенджер Telegram.

**Технологии:**
- Rust + teloxide (или Python + aiogram)
- Long polling или Webhooks
- Inline keyboards для навигации
- Markdown/HTML форматирование

**Команды:**

1. **/start** — Приветствие, инструкция
2. **/search <name>** — Поиск игрока
3. **/analyze <name>** — Полный AI анализ
4. **/compare <player1> <player2>** — Сравнение
5. **/team <abbreviation>** — Состав команды
6. **/news <player>** — Последние новости
7. **/subscribe** — Подписка на уведомления
8. **/settings** — Настройки бота

**Inline режим:**
- Пользователь вводит `@hockeybot McDavid`
- Показывает quick results
- Клик → детальная информация

**Flow: Анализ игрока**
```
User: /analyze Ovechkin
  ↓
Bot: Search player via MCP: search_players(query="Ovechkin")
  ↓
Bot: Show inline keyboard with top 5 results
  ↓
User: Clicks "Alexander Ovechkin (WSH)"
  ↓
Bot: Call MCP: analyze_player(name="Alexander Ovechkin", focus="all")
  ↓
Bot: Format response with emojis:
📊 *Александр Овечкин* | LW | 38 лет | WSH

🎯 *Сильные стороны:*
• Элитный бросок с хода (Sh% 13.2, топ-5 NHL)
• Позиционирование в большинстве (PP G/60: 2.8)
• Лидерские качества (капитан с 2010)
```

```sql
CREATE VIEW v_player_career_summary AS
SELECT 
    p.id,
    p.nhl_player_id,
    p.full_name,
    p.position,
    p.current_team_abbreviation,
    COUNT(DISTINCT pss.season) as seasons_played,
    SUM(pss.games_played) as career_games,
    SUM(pss.goals) as career_goals,
    SUM(pss.assists) as career_assists,
    SUM(pss.points) as career_points,
    AVG(pss.points_per_game) as career_ppg,
    MAX(pss.points) as best_season_points,
    (SELECT season FROM player_season_stats 
     WHERE nhl_player_id = p.nhl_player_id 
     ORDER BY points DESC LIMIT 1) as best_season
FROM players p
LEFT JOIN player_season_stats pss ON p.nhl_player_id = pss.nhl_player_id
WHERE pss.league = 'NHL'
GROUP BY p.id, p.nhl_player_id, p.full_name, p.position, p.current_team_abbreviation;
```

#### fn_update_player_stats

```sql
CREATE OR REPLACE FUNCTION fn_update_player_stats(
    p_nhl_player_id INTEGER,
    p_season TEXT,
    p_raw_data JSONB
)
RETURNS INTEGER AS $$
DECLARE
    v_player_id INTEGER;
    v_stats_id INTEGER;
BEGIN
    -- Get or create player
    SELECT id INTO v_player_id 
    FROM players 
    WHERE nhl_player_id = p_nhl_player_id;
    
    IF v_player_id IS NULL THEN
        RAISE EXCEPTION 'Player not found: %', p_nhl_player_id;
    END IF;
    
    -- Upsert stats
    INSERT INTO player_season_stats (
        nhl_player_id, player_id, season, raw_api_data,
        games_played, goals, assists, points,
        calculated_at
    )
    VALUES (
        p_nhl_player_id, v_player_id, p_season, p_raw_data,
        (p_raw_data->>'games')::INTEGER,
        (p_raw_data->>'goals')::INTEGER,
        (p_raw_data->>'assists')::INTEGER,
        (p_raw_data->>'points')::INTEGER,
        NOW()
    )
    ON CONFLICT (nhl_player_id, season, team_abbreviation)
    DO UPDATE SET
        games_played = EXCLUDED.games_played,
        goals = EXCLUDED.goals,
        assists = EXCLUDED.assists,
        points = EXCLUDED.points,
        raw_api_data = EXCLUDED.raw_api_data,
        calculated_at = NOW()
    RETURNING id INTO v_stats_id;
    
    -- Recalculate per-game averages
    UPDATE player_season_stats
    SET 
        points_per_game = CASE 
            WHEN games_played > 0 THEN ROUND(points::DECIMAL / games_played, 2)
            ELSE 0
        END,
        updated_at = NOW()
    WHERE id = v_stats_id;
    
    RETURN v_stats_id;
END;
$$ LANGUAGE plpgsql;
```

```rust
struct Claims {
```

```yaml
        annotations:
          summary: "High latency detected"
          description: "P95 latency is {{ $value }}s"

      # Database connection pool exhausted
      - alert: DatabaseConnectionsExhausted
        expr: pg_stat_activity_count / pg_settings_max_connections > 0.9
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "Database connections nearly exhausted"

      # AI service down
      - alert: AIServiceDown
        expr: up{job="openclaw"} == 0
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "OpenClaw AI service is down"

      # n8n workflow failures
      - alert: WorkflowFailures
        expr: rate(n8n_workflow_errors_total[5m]) > 0.1
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "High n8n workflow failure rate"

      # Disk space low
      - alert: DiskSpaceLow
        expr: (node_filesystem_avail_bytes / node_filesystem_size_bytes) < 0.1
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "Disk space running low"

      # Memory usage high
      - alert: MemoryUsageHigh
        expr: (node_memory_MemTotal_bytes - node_memory_MemAvailable_bytes) / node_memory_MemTotal_bytes > 0.9
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Memory usage above 90%"
```
